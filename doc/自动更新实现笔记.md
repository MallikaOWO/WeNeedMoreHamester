# 自动更新功能 — 实现笔记

## 架构概述

更新功能涉及两个 iframe 和主窗口三个上下文：

| 上下文 | 生命周期 | 用途 |
|--------|---------|------|
| **主窗口** (window.top) | 始终存活 | 承载 SillyTavern 和 `window.TavernHelper` |
| **脚本 iframe** (TH-script--*) | 角色重载时销毁重建 | 运行角色卡脚本（MVU、变量结构、自动更新等） |
| **消息 iframe** (TH-message--*) | 角色重载时存活 | 运行 React 前端 UI |

## 核心坑点：脚本 iframe 在更新时会被销毁

`importRawCharacter(name, blob)` 将新的角色卡 PNG 导入酒馆。导入成功后，SillyTavern 触发角色重载，**销毁并重建所有脚本 iframe**。

这意味着：
- 在脚本 iframe 中 `await importRawCharacter(...)` 的 Promise **永远不会 resolve**，因为执行环境已被销毁
- 之后的所有代码（`updateCharacterWith`、`toastr`、`return true` 等）都不会执行
- 如果从 React 前端通过 UpdateService 桥接调用脚本 iframe 的 `installUpdate()`，React 侧收到的 Promise 也会永久挂起

### 错误方案历程

1. **方案一：脚本 iframe 中 await importRawCharacter + 后续操作** — 后续代码永远不执行
2. **方案二：React 侧 fire-and-forget + 轮询版本号** — 轮询能检测到新版本，但实际效果不稳定
3. **方案三：通过 UpdateService 桥接** — 本质和方案一/二相同，Promise 挂起问题无法解决

### 正确方案：从消息 iframe 直接调用 TavernHelper

消息 iframe 在角色重载时**不会被销毁**。`@types/function/` 下的所有酒馆助手 API 都导出在 `window.TavernHelper` 上，消息 iframe 可通过 `window.top.TavernHelper` 直接调用。

因此，完全绕过脚本 iframe，在 React 代码中直接完成：下载 → 世界书备份 → importRawCharacter。

```typescript
// update.ts — 从消息 iframe 直接调用
export async function installUpdate(downloadUrl: string): Promise<void> {
  const resp = await fetch(downloadUrl, { cache: 'no-cache' });
  const blob = await resp.blob();
  const th = (window.top as any).TavernHelper;
  await th.importRawCharacter('鼠鼠天堂', blob);
}
```

这样 `await` 可以正常工作，无需轮询或 fire-and-forget。

## 第二个坑：importRawCharacter 后 UI 不刷新

`importRawCharacter` 成功后：
- **后端数据已更新**：通过 `getCharacter()` API 读取可以看到新的版本号、脚本路径等
- **SillyTavern 前端缓存已更新**：`SillyTavern.getContext().characters` 中的数据也是新的
- **但 UI 面板不自动刷新**：如果脚本编辑面板、正则面板等已打开，显示的仍是旧数据

**用户必须手动刷新酒馆页面（F5）** 才能看到并使用新版本的脚本、正则、世界书等。

重要：刷新后可以在**同一个聊天记录**中继续游玩（变量结构未变的前提下）。游戏存档（MVU 变量 stat_data）存在聊天消息里，不受角色卡更新影响。

## 第三个坑：replaceCharacter 是完全替换

`replaceCharacter(name, { version: '1.0' })` 看似只更新版本字段，实际上是**完全替换**整个角色卡数据，只保留传入的字段，其余全部清空。

参考脚本 `综合更新脚本.js` 中 `replaceCharacter(name, { version })` 这行虽然存在，但因为脚本 iframe 已被 `importRawCharacter` 销毁，它实际从未执行。如果真的执行了，会把角色卡数据清空。

安全的部分更新应使用 `updateCharacterWith(name, char => { char.version = ver; return char; })`。

但在当前方案中，不需要手动更新版本号——发行 PNG 自身已包含正确的版本号（在 release.sh 构建时写入）。

## URL 方案

| 资源 | 域名 | 原因 |
|------|------|------|
| manifest.json | `raw.githubusercontent.com` | 文本文件，无 CDN 缓存，确保拿到最新版本信息 |
| 角色卡 PNG | `media.githubusercontent.com` | Git LFS 大文件；`raw.githubusercontent.com` 对 LFS 只返回指针文件 |

manifest.json 请求时附加 `?t=timestamp` 参数并设置 `cache: 'no-store'` 防止浏览器缓存。

## 文件清单

| 文件 | 职责 |
|------|------|
| `src/鼠鼠天堂/界面/主界面/update.ts` | manifest 获取、版本比较、TavernHelper 直接调用（下载+导入+世界书备份+toast） |
| `src/鼠鼠天堂/界面/主界面/update-check.ts` | 轻量更新检查（模块级缓存，供 App 齿轮红点使用） |
| `src/鼠鼠天堂/界面/主界面/views/Settings.tsx` | 设置页 UI：版本信息、更新按钮、更新日志、关于 |
| `src/鼠鼠天堂/脚本/自动更新/index.ts` | 脚本 iframe 中的 UpdateService（历史遗留，当前未被 React 侧使用） |
| `manifest.json`（仓库根目录） | 版本清单，包含各版本的路径、日期、changelog |

## 调试经验

- SillyTavern 的 MCP 调试中，`console.log` 可能不被捕获，优先用 `return` 值或 `console.error`
- 从主窗口 JS 控制台执行代码验证 API 行为比从 iframe 内验证更可靠
- 测试更新时只用本地版本 → 最新发行版，不要在发行版之间切换（脚本路径不同会引入额外变量）
